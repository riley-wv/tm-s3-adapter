# AGENTS.md

## Product Context

`tm-s3-adapter` is a VPS-first service that lets users connect cloud storage and expose it as an SMB share that macOS Time Machine can use as a backup destination.

Supported storage patterns include:

- S3 and S3-compatible object storage
- Google Drive and OneDrive through rclone
- Generic rclone remotes
- Local disk paths for baseline validation and fallback

The main goal is dependable Time Machine backups over SMB, with cloud storage acting as the durable backend. Favor reliability, compatibility, and safe operational behavior over unnecessary abstraction.

## What Matters Most

- Time Machine compatibility is the core product requirement. Changes that affect Samba share semantics, metadata streams, locking, quotas, or mount consistency should be treated as high risk.
- Local mode is the simplest known-good path. When debugging cloud-backed issues, keep the option to validate behavior against a local disk first.
- Secrets must never leak. Access keys, secret keys, admin passwords, API tokens, SMB passwords, and SFTP credentials should be redacted from logs, errors, and UI responses.
- The default network exposure model is intentional. Dashboard, API, SMB, and SFTP are bound to `127.0.0.1` in `docker-compose.yml`; avoid widening exposure unless the task explicitly requires it.

## Repository Map

- `src/vpsd/index.mjs`: main daemon, admin/public API, metadata orchestration, settings resolution, static asset serving
- `src/vpsd/cloudMountManager.mjs`: rclone/FUSE mount lifecycle and cache policy handling
- `src/vpsd/sambaManager.mjs`: Samba user management and generated share configuration
- `src/vpsd/sftpManager.mjs`: SFTP service management
- `src/shared/*.mjs`: shared HTTP helpers, command execution, storage, filesystem utilities
- `web/dashboard/`: source for the admin dashboard
- `web/vps-public/`: generated static dashboard output; rebuild it instead of editing files here directly
- `deploy/`: Samba and Cloudflare-related deployment templates
- `docs/`: setup, configuration, API, and operational guidance

## Architecture Notes

- The app runs as a single container for the main service plus Postgres in Compose.
- Postgres-backed settings are required for setup/settings persistence.
- Cloud-backed disks rely on rclone mounts and host FUSE support (`/dev/fuse`).
- SMB shares are generated by the app, not hand-managed.
- The dashboard is a Next.js app exported to static files and served by the Node service.

## Critical Invariants

- Preserve Samba Time Machine settings unless the task explicitly changes product behavior. In particular, be careful with:
  - `fruit:time machine = yes`
  - `vfs objects = catia fruit streams_*`
  - durable handles and locking-related settings
  - quota behavior via `fruit:time machine max size`
- Do not edit generated Samba output or generated dashboard export as the source of truth when the real change belongs in source code.
- Keep config changes backward-compatible where reasonable. If you add or rename env vars, update `.env.example`, docs, and any code that reads them.
- Cloud mount failures should fail clearly and safely. Avoid changes that make mount issues silent or that block local-mode disks.
- Be cautious with shell command construction. This project invokes system tools like `rclone`, `smbpasswd`, and Samba reload commands, so input normalization and redaction matter.

## Working Rules For Agents

- Start by reading the relevant docs before changing behavior that affects setup, configuration, or operations.
- Prefer fixing the source in `web/dashboard/` and rebuilding instead of modifying `web/vps-public/`.
- Prefer changes that keep the admin UI, API, docs, and runtime behavior aligned.
- If you touch mount/provider logic, consider S3-compatible providers, not just AWS S3.
- If you touch storage or backup behavior, think through both SMB and SFTP paths.
- If you change persistence or settings shape, account for existing metadata and saved settings.

## Validation Expectations

Run the smallest relevant checks for the area you changed:

- `npm test` for shared logic and utility changes
- `npm run dashboard:build` for dashboard/source changes
- `npm run docker:up` when validating full container/runtime behavior

When validating Time Machine-related behavior, prefer this order:

1. Confirm local-mode disk behavior works.
2. Confirm Samba config/share generation still matches expected Time Machine semantics.
3. Confirm cloud-mount behavior, cache settings, and mount error handling.

## File-Specific Guidance

- Do not hand-edit `web/vps-public/` unless the task is explicitly about generated output inspection.
- Treat `data/` as runtime state, not source.
- Keep docs current when changing setup flow, environment variables, ports, storage providers, or security posture.

## Definition Of A Good Change

A good change in this repo:

- improves or preserves Time Machine compatibility
- does not expose secrets
- does not weaken the default local-only network model by accident
- keeps local disks and cloud mounts understandable to operators
- updates docs/examples when behavior or configuration changes
